# Coverage Review

Review coverage evidence for `aim` using the bound repository coverage workflow.

This skill is the `aim` productized coverage counterpart. It does not assume diff-line automation is available. Instead, it stays inside the repository-supported coverage signals that `aim` can actually generate today.

## When to Use

- When a review needs explicit coverage evidence
- When test changes are substantial and regression protection is uncertain
- When verifying whether follow-up fixes preserved or improved coverage
- Before completion if coverage is a real gate for the change

## Bound Command

- default coverage command:
  - `make gtest`

Related commands:

- `make gtest-build`
- `make gtest-run`
- `make html`
- `script/lcov-clean.sh`
- `script/lcov-make.sh`
- `script/lcov-test.sh`
- `script/lcov-html.sh`

Command safety note:

- use `make gtest` as the default evidence path for this skill
- treat `make html` and `script/lcov-*.sh` as supplemental drilldown only when the environment is ready
- do not replace this workflow with `script/dist/patch_aim.sh`, `script/dist/dist_aim.sh`, or `script/ofrelease.sh` just to obtain a stronger-sounding result

## Scope

This skill measures repository-supported coverage artifacts produced by the default GoogleTest flow and optional deeper drilldowns.

Repository-aware boundary:

- current proven coverage scope:
  - GoogleTest binaries executed by `make gtest`
  - summary data emitted under `test/unit/gtest/report/`
  - module HTML coverage generated by `make html` when gcov data already exists
- optional but environment-dependent scope:
  - lcov HTML generated through `script/lcov-*.sh`
- outside the first review boundary:
  - packaging and distribution scripts under `script/dist/`
  - release flow under `script/ofrelease.sh`
  - runtime boot assumptions required by `script/lcov-test.sh`

Reason:

- the confirmed default coverage command is `make gtest`, which already produces the most reliable repo-native summary evidence
- deeper lcov flow depends on runtime services and environment-specific setup

It can reliably report:

- whether `make gtest` completed
- whether report files were produced under `test/unit/gtest/report/`
- top-level GoogleTest and global coverage summaries when present
- whether changed areas appear covered by matching gtest trees or nearby tests

It must not claim:

- exact diff-line coverage unless the repository provides a trustworthy diff-aware report
- a policy threshold that is not explicitly confirmed for `aim`
- that every changed path in the repo was included in the exercised test scope

## Workflow

### Step 1: Decide Whether Coverage Review Is Needed

Use it when:

- production logic changed
- tests changed materially
- the reviewer or user asked for coverage evidence
- completion depends on stronger regression confidence

Skip or reduce scope when:

- docs-only changes
- comment-only changes
- metadata-only changes

### Step 2: Run Coverage Generation

Run the bound default coverage command for `aim`.

If deeper drilldown is needed and the environment supports it, optionally run:

- `make html`
- the `script/lcov-*.sh` sequence

If supplemental commands cannot run because required environment values or runtime services are missing, report that blocker explicitly before making any claim about those paths.

### Step 3: Inspect Coverage Outputs

Look for generated outputs under:

- `test/unit/gtest/report/`
- `test/unit/gtest/report/xml/`
- `test/unit/gtest/report/log/`
- `test/unit/gtest/report/meta/`
- module HTML coverage paths produced by `make html`
- `$LCOV_OUTDIR/aim/` when the lcov workflow was intentionally run

Prefer repository-generated outputs over hand-written interpretation.

Collect:

- report generation success or failure
- top-level coverage percentages when available
- modules or surfaces with obviously weak coverage
- whether changed areas appear covered by matching tests or reported files

### Step 4: Compare Against Change Scope

Use local git diff and changed files to answer:

- which production files changed
- which tests changed
- whether the test additions map to the changed behavior
- whether the coverage signal is broad, narrow, or missing
- whether the changed files were actually inside the exercised gtest or lcov scope

Do not overstate precision. If the signal is only module-level, say so explicitly.

If the change is mainly in generated data, packaging scripts, release flow, or repo areas not exercised by the current coverage path, report the boundary instead of forcing a misleading conclusion.

### Step 5: Write Coverage Review

Write the result to:

- `agent/<topic>/coverage_review.md`

Suggested shape:

```markdown
# Coverage Review

## Command
- coverage:

## Report Status
- generated:
- report paths:

- summary:
- weak areas:

## Change-Scope Comparison
- changed production files:
- changed test files:
- mapping assessment:

## Conclusion
- confidence level:
- gaps or follow-up:
```

## Rules

- Separate report facts from reviewer inference.
- Do not pretend `make gtest` or optional lcov output is full repository or diff-line coverage.
- Do not invent a required percentage threshold unless the product policy is confirmed.
- If coverage generation fails, report the failure and the likely blocker.
- If the change is outside the exercised scope, report the coverage boundary explicitly instead of forcing a misleading conclusion.
