## 2. Phase 1 — 메타 결정

### Phase 1-a. SSoT 정책 확정 ✅ (2026-04-21)

#### Problem
3건의 skill gap 보고서에서 공통 패턴: 규칙 상세 내용이 여러 저장소(SKILL.md abbreviated summary + memory 본문 + guide + governance)에 **중복 산재**. 결과:
- 단일 소스 식별 불가 → 어디를 기준으로 할지 에이전트가 혼란
- 한 곳만 갱신되고 다른 곳은 낙후 → 규칙 불일치
- 에이전트가 "abbreviated summary만 보고" 작성 → 상세 규칙 누락
  - 실사례: IMS#353696 gtest canonical 누락, MR !588 verbatim 누락

**결정할 것**: 어느 저장소를 canonical source로 둘지, 중복을 어디까지 허용할지.

#### Options considered

| 옵션 | 내용 | 문제점 |
|---|---|---|
| A. 모든 규칙을 governance로 집중 | 팀 공유 최대화 | governance 비대화. 에이전트 절차(skill)까지 여기로 몰아가는 건 부자연스러움 |
| B. 모든 규칙을 skill guide로 집중 | skill 진입점 통일 | 코드/빌드 함께 이동해야 할 governance 규칙이 guide에 고립됨 |
| C. 모든 규칙을 memory로 집중 | 빠른 갱신 | 팀 공유 불가(memory는 개인), git 추적 없음 |
| D. 저장소 성격별 분할 + 도메인별 SSoT 지정 | 각 저장소 장점 활용 | SSoT 매핑 정책 필요, 복잡도 있음 |

#### Decision

**1) 판단 프레임: "Who owns the rule? + Lives with what?"**
- 규칙이 **코드/빌드 산출물 구조**와 함께 움직이면 → **governance** (코드 트리에 동거)
- 규칙이 **에이전트가 수행하는 절차/체크리스트**면 → **skill guide**
- 규칙이 **세션별 경험 누적/개인 히스토리**면 → **memory**
- **SKILL.md는 SSoT 아님** → routing 포인터 + workflow 다이어그램만

**2) 저장소 분류 (6개)**

| 저장소 | 위치 | 성격 | 갱신 빈도 |
|---|---|---|---|
| **governance** | `AGENTS.md` / `AGENTS.override.md` (각 zone) | git 추적, 팀 공유 | 낮음 |
| **skill guide** | `.claude/skills/<skill>/<domain>-guide.md` | git 추적, 팀 공유 | 중 |
| **SKILL.md** | `.claude/skills/<skill>/SKILL.md` | git 추적, 팀 공유 | 중 |
| **memory** | `~/.claude/projects/.../memory/*.md` | 개인, 외부 | 높음 |
| **.gitmessage** | 프로젝트 루트 | git commit 시 에디터 자동 로드 | 낮음 |
| **.gitlab/merge_request_templates/default.md** | 프로젝트 루트 | GitLab이 MR 생성 시 자동 로드 | 낮음 |

**3) 도메인별 SSoT 매핑**

| 도메인 | SSoT | 보조 저장소 | 비고 |
|---|---|---|---|
| **gtest canonical format** (HP_/E_, 섹션 마커, Suite comment, body 주석) | `aim/test/unit/gtest/AGENTS.override.md` | memory (단축형 트리거), test-driven-development-aim SKILL.md (포인터) | governance가 SSoT, memory/skill은 참조 |
| **Module naming** (`lib<MODULE>` vs `<MODULE>`) | `aim/src/<zone>/AGENTS.override.md` (이미 정의됨) | `gitlab-guide.md` (링크만) | 기존 governance 활용 |
| **MR description 작성 절차** (섹션 순서, verbatim 요구, 적신호) | `.claude/skills/writing-documents-aim/gitlab-guide.md` | finishing-a-development-branch-aim SKILL.md (포인터) | guide가 SSoT |
| **MR template 뼈대** (섹션 구조) | `aim/.gitlab/merge_request_templates/default.md` | `gitlab-guide.md` (참조) | 실행환경 자동 로드 파일이 뼈대 SSoT |
| **commit message 형식** | `aim/.gitmessage` | `AGENTS.md` (포인터) | 실행환경 자동 로드 파일이 SSoT |
| **Manual 작성 절차** | `.claude/skills/writing-documents-aim/manual-guide.md` | 독립 | guide가 SSoT (Phase 3) |
| **Self-review gate 공통 패턴** | 각 유형 A guide의 `## Self-review checklist (적신호)` 섹션 | `aim/AGENTS.md` (1줄 슬로건), 유형 A SKILL.md (Workflow Diagram 게이트 노드) | Phase 1-c 확정 |

**4) 중복 제거 원칙**
1. **SSoT 외부 기재 금지**. 단 "요약 1~3줄 + SSoT 링크"는 허용 (skill 진입점 명확화 목적)
2. **memory는 단축형 포인터로 전환**: canonical 규칙 상세 내용 제거 → 표준 3필드 (Phase 1-b Decision D)
   - 삭제 대신 **트리거 역할** 유지: `MEMORY.md` 인덱스가 자동 로드되어 Plan 단계/메인 에이전트의 선제 Read 유도
3. **실행환경 자동 로드 파일**(.gitmessage, MR template)은 **내용의 SSoT**, **작성 절차/체크리스트**는 guide의 SSoT — 역할이 달라서 **공존 가능**

#### Rationale

- **옵션 D 선택 이유**: 4개 저장소는 각각 서로 다른 접근 경로/수명/공유 범위를 가짐. 한 곳에 몰면 저장소별 장점(git 추적 vs 개인 빠른 갱신 vs 코드 동거 등)이 손실됨. 각 저장소의 본래 성격에 맞는 규칙을 두는 게 유지보수 효율적.
- **판단 프레임을 "Who owns + Lives with"로 둔 이유**: 규칙의 성격(팀 규칙/절차/경험)과 동반 대상(코드 트리/skill 체계/개인 히스토리)을 함께 봐야 중복 없이 한 곳을 지정 가능. 단일 축(예: 갱신 빈도)만 보면 같은 빈도의 규칙들이 서로 다른 저장소로 가는 현상 발생.
- **memory 완전 삭제 대신 단축형 포인터로 전환한 이유**: `MEMORY.md` 인덱스가 세션 시작 시 자동 로드되어 **선제 트리거** 역할을 함. 완전 삭제 시 Plan 단계에서 에이전트가 SSoT 존재를 인지하지 못할 위험 (AGENTS.override.md는 zone 진입 후에야 참조됨 → Plan 단계에서는 zone 미진입 상태일 수 있음).
- **실행환경 자동 로드 파일(.gitmessage, MR template)을 별도 SSoT로 인정한 이유**: 이들은 **사람이 채우는 템플릿 + 실행환경이 자동 로드** 성격. guide와 역할이 달라 공존 가능 (template 뼈대는 default.md, 작성 절차/적신호는 gitlab-guide).

#### 확정 시점
- 2026-04-21, 사용자 승인

### Phase 1-b. SKILL.md ↔ guide/memory 역할 분담 ✅ (2026-04-21)

#### Problem
Phase 1-a에서 저장소 성격별 SSoT 분할을 결정했으나, 실제 적용 시 9개 세부 질문 발생:

- **Q1**. 모든 SKILL.md에서 규칙 상세를 빼는 게 가능한가? (일부 skill은 자체 완결적일 수 있음)
- **Q2**. "Read 의무" 표기를 몇 단계로 둘지
- **Q3**. memory 단축형 포인터의 표준 필드 구성
- **Q4**. Context window 부담 관리 기준 (lines vs Read 단계 수)
- **Q5**. 슬로건 수준의 중복 허용 여부
- **Q6**. Skill 간 상호 참조 (Read vs Invoke) 구분
- **Q7**. 실행환경 자동 로드 파일의 Read 의무 표기 방식
- **Q8**. 방법론형 skill에도 self-review gate 적용할지
- **Q9**. SSoT 간 중복 허용 수준

#### Options considered (Q별 요지)

**Q1. SKILL.md 규칙 상세 제거 범위**
- α. 모든 skill에 일괄 적용 → 방법론형 skill은 본문이 사라져 껍데기가 됨
- β. 유형별 차등 적용 (참조형만) → 실제 skill 파일 구조 확인 후 분류 필요

**Q2. Read 의무 단계**
- α. 3단계 (✅ 필수 / 🔶 권장 / 공백) → 🔶 모호 판단을 에이전트에 위임 → 결국 안 Read 방향
- β. 2단계 (✅ / 공백) → 조건부 필수는 ✅ + 비고에 조건

**Q3. memory 템플릿 필드**
- α. 4필드 (SSoT/Read의무/누락이력/한줄요약) → "Read의무"와 "한줄요약" 의미 중복
- β. 3필드 (SSoT/When/Why) → 간결, Why에 이력 누적

**Q4. Context window 부담 기준**
- α. Lines 기준 (~1000) → 성숙도 따라 자연 증가, 임의적
- β. Read 단계 수 기준 (2단계 이하) → context miss 주원인, 직접적
- γ. β 주 기준 + Lines 보조 기준 (SSoT 500 lines) → 둘의 장점 결합

**Q6. Skill 간 참조**
- α. Invoke/Read 미구분 → 현재 CLAUDE.md 워크플로우 체인에서 혼란
- β. Invoke (Skill 도구 호출) vs Read (문서만 참조) 용어 분리

**Q7. 자동 로드 파일 Read 의무**
- α. ✅ 필수 표기 → 자동 로드되는 파일을 굳이 에이전트가 Read하면 중복
- β. 공백 표기 + "SSoT 역할만 / 자동 로드" 비고

**Q8. 방법론형 skill의 self-review gate**
- α. 유형 무관 일괄 적용 → 외부 산출물이 없는 skill에는 과부하
- β. 유형 A만 적용, 유형 B는 자체 검증 패턴 유지

**Q9. SSoT 간 중복 허용 수준**
- α. 완전 금지 → 독자가 한 파일만 보면 맥락 파악 곤란
- β. Phase 1-a와 동일 원칙(요약 1~3줄 + 링크 허용, 상세 복제 금지)

#### Decision

**A. Skill 3유형 분류 (실제 파일 구조 검토 기반)**

| 유형 | 특징 | 해당 skill | 규칙 처리 |
|---|---|---|---|
| **A. 참조형** | 규칙이 외부 저장소/채널별 분리 가능 | writing-documents-aim, test-driven-development-aim, finishing-a-development-branch-aim | SKILL.md는 포인터, 규칙은 SSoT |
| **B. 방법론형** | skill 자체가 사고/행동 패턴 본체 | brainstorming-aim, systematic-debugging-aim, issue-analysis-aim, verification-before-completion-aim, using-aim-harness, dispatching-parallel-agents-aim, refactoring-catalog, vulnerability-patterns, using-feature-branches-aim, executing-plans-aim, completing-patch-aim, receiving-code-review-aim, requesting-code-review-aim | SKILL.md가 본문, 500 lines 초과 시 내부 분할 |
| **C. 오케스트레이션형** | 메인 + 서브에이전트 프롬프트 | code-reviewer-aim, subagent-driven-development-aim, writing-plans-aim, brainstorming-aim, writing-skills-aim | SKILL.md는 워크플로우 허브, 프롬프트는 별도 파일 (대부분 이미 구현됨) |

**B. SKILL.md 표준 구조 (유형 A 대상, 5개 블록)**
1. Frontmatter — name, description, when to invoke
2. Purpose — 1~3줄, skill의 존재 이유
3. Workflow Diagram — 단계 흐름
4. Routing Table — 아래 C 참조
5. Integration — 다른 skill/체인과의 연결

**C. Routing Table 표준 형식**

| 상황 | SSoT | Read 의무 | 비고 |
|---|---|:---:|---|
| (진입 상황 1) | <SSoT 경로> | ✅ 또는 공백 | 조건부 필수일 경우 조건 명시 |

**Read 의무 2단계**: ✅ 필수 / (공백) 불필요. 🔶 권장 제거.

**D. memory 파일 표준 3필드**

```markdown
---
name: {{}}
description: {{트리거 역할 + SSoT 링크}}
type: feedback / reference
---

**SSoT**: <절대경로>
**When**: 언제 Read하는가 (트리거 조건)
**Why**: 이 메모리가 존재하는 이유 (최초 누락 이력 + 규칙 방향 한두 줄, 반복 누락 시 누적)
```

**E. Context window 부담 관리**
- **주 기준**: Read 단계 **2단계 이하** (SKILL.md → SSoT). SSoT가 또 다른 SSoT를 **필수** Read로 요구하면 3단계 → 재구조화 대상. 선택 Read는 단계 수 미포함.
- **보조 기준**: SSoT 단일 파일 500 lines 초과 시 분할 검토 (섹션별 별도 파일)

**F. 슬로건 허용**
SSoT의 1줄 슬로건 수준 핵심 원칙은 SKILL.md Purpose에 반복 기재 허용 (재확인 포인트로 기능).

**G. Skill 간 참조 구분**
- **Invoke**: 다른 skill을 Skill 도구로 호출 (워크플로우 이동)
- **Read**: 해당 skill의 guide 파일만 Read (문서 참조)
- Routing Table 비고에 `Invoke` 또는 `Read` 중 어느 쪽인지 명시

**H. 실행환경 자동 로드 파일의 Read 의무**
- 자동 로드 → 에이전트 Read 불필요 → **Read 의무 = 공백**, 비고에 "SSoT 역할만 / 자동 로드" 표기
- 해당 파일의 **작성 절차**는 guide SSoT에서 커버, 이 guide는 ✅ 필수

**I. 방법론형(유형 B) skill의 self-review gate**
- 유형 B는 외부 산출물이 없는 경우가 있음 (예: systematic-debugging-aim은 사고 방법론)
- **self-review gate는 유형 A에만 강제** 적용. 유형 B는 자체 검증 패턴 유지.

**J. SSoT 간 중복 허용 수준**
- SSoT → SSoT 간에도 Phase 1-a의 "요약 1~3줄 + 링크 허용, 상세 복제 금지" 원칙 적용
- 예: Module naming은 `src/lib/AGENTS.override.md`가 SSoT, `gitlab-guide.md`에서 Squash commit 설명 시 1~3줄 요약 + 링크만

#### Rationale

- **유형 분류를 한 이유 (A)**: 모든 skill에 일괄 "SSoT 분리" 강제는 방법론형 skill의 본문을 사라지게 만듦 (brainstorming-aim의 설계 방법론을 외부로 빼면 어디로?). `skills/` 디렉토리 실측 결과 3유형이 명확히 구분됨 — 차등 적용이 현실적.
- **Read 의무 2단계로 축소한 이유 (C)**: 3단계의 🔶는 "알아서 판단"을 에이전트에 위임 → 실전에서 Read 안 하는 방향으로 기움. 조건부 필수를 명시적으로 ✅ + 비고로 전환하면 모호성 제거.
- **memory 3필드 이유 (D)**: "Read 의무"와 "한 줄 요약"이 의미 중복(둘 다 SSoT를 가리킴). When/Why로 통합하면 간결. Why에 누락 이력을 누적 → 시간이 지나도 별도 메모리 생성 없이 관리 가능.
- **Read 단계 수를 주 기준으로 선택한 이유 (E)**: Lines는 skill/SSoT 성숙도에 따라 자연 증가하므로 절대 기준 부적합. 단계 수는 **context miss의 실제 원인** (에이전트가 중간 단계에서 Read를 멈춤) — 직접적 지표. Lines는 보조로 남겨 SSoT 분할 시점 판단에 활용.
- **Skill 간 Invoke/Read 분리 이유 (G)**: 현재 CLAUDE.md 워크플로우 체인이 skill → skill 이동(Invoke)으로 되어 있는데, 여기에 Read(문서만 참조)가 섞이면 혼란. 용어 분리로 의도 명확화.
- **자동 로드 파일 Read 의무 공백 이유 (H)**: git/GitLab이 이미 템플릿으로 로드 → 에이전트가 중복 Read 불필요. 단 **작성 절차**는 별도 SSoT(guide)이므로 그쪽에 ✅.
- **self-review gate를 유형 A만 적용한 이유 (I)**: 유형 B는 "외부 산출물" 개념이 없거나 다름 (debugging은 코드 수정이지만 checklist 자체가 방법론임). 일률 적용 시 과부하이며 방법론의 자체 흐름 방해.
- **SSoT 간 중복 1~3줄 허용 이유 (J)**: 완전 금지 시 독자가 한 파일만 봐서는 전체 맥락 파악 곤란. 링크 따라가는 부담 완화 차원. 단 상세 복제는 금지(갱신 불일치 위험) — Phase 1-a 원칙과 동일 선상.

**B. 편입 위치**

**D. 통과 증거 기록**
- α. 암묵적 (통과했다고 주장하면 OK) → 검증 불가
- β. 명시적 (PR description에 "checklist passed" 기재) → 모든 산출물에 요구 시 과부하
- γ. 기본 암묵적 + **위반 항목만 명시적** (DONE_WITH_CONCERNS에 기재) → 실용적 균형

**E. Skill Gap 자발 보고와의 경계**
- α. 둘을 같은 채널로 통합 → 성격 다름 (개별 작업 실패 vs skill 체계 결함)
- β. 분리: 체크리스트 위반 = DONE_WITH_CONCERNS / 체크리스트 자체 오류·누락 = Skill Gap

#### Decision

**A. 구체 형태: 혼합 (γ)**
체크리스트를 기본으로 하고, 자동화 가능한 항목은 guide에 `grep` / `script` 명령 병기.

**B. 편입 위치: ε (3층 구조)**

| 계층 | 위치 | 역할 |
|---|---|---|
| **SSoT** | 각 guide의 `## Self-review checklist (적신호)` 섹션 | 체크리스트 상세 + 자동화 명령 |
| **Skill 워크플로우** | 유형 A SKILL.md의 Workflow Diagram | "작성 → **self-review gate** → PUT" 3노드 명시 |
| **전역 원칙** | `aim/AGENTS.md` (또는 `CLAUDE.md`) | 1줄 슬로건: "외부 산출물은 PUT 전 해당 guide의 Self-review checklist 통과" |

**C. 트리거 시점**
외부 산출물 제출 **직전** (PUT / commit / push). 작성 완료 직후가 아니라 제출 직전 — 직전이 마지막 방어선.

**D. 통과 증거 기록**
- **기본**: 암묵적 통과 (모든 항목 명시 요구 안 함)
- **위반 발견 시**: DONE_WITH_CONCERNS에 위반 항목과 대응 기재
- **PR description 등 최종 산출물**: 자체 형식에 이미 포함된 검증(예: Test 섹션 verbatim)은 통과 증거로 간주

**E. Skill Gap 자발 보고와의 경계**

| 상황 | 보고 채널 | 형식 |
|---|---|---|
| 체크리스트 **위반** (내가 놓쳤다) | DONE_WITH_CONCERNS | `[Check Fail] <checklist item>: <상황>` |
| 체크리스트 **자체 누락/오류** (guide에 없어서 놓쳤다) | Skill Gap | `[Skill Gap] <skill>: <내용>` |
| 둘 다 | 각각 분리 보고 | — |

**F. 적용 범위 (Phase 1-b Decision I 재확인)**
- **유형 A skill만 강제** (writing-documents-aim, test-driven-development-aim, finishing-a-development-branch-aim 등)
- 유형 B (방법론형)는 제외 — 외부 산출물 개념이 없거나 자체 검증 패턴을 가짐

**G. 적신호 섹션 표준 형식 (guide SSoT)**

```markdown
## Self-review checklist (적신호)

PUT/commit/push **직전** 아래 항목을 확인한다. 위반 시 재작성 후 재제출.

- [ ] <체크 항목 2>

1. 재작성 후 재제출, 또는
2. 부득이한 경우 DONE_WITH_CONCERNS 리포트에 `[Check Fail] <항목>: <상황>` 기재
```

**H. 적신호 목록 갱신 원칙**
사용자 지적이 새로 발생 → 해당 guide의 적신호 섹션에 **항목 누적**. 누적은 skill 수정이 아니라 guide 갱신 (SSoT 정책 유지). **Skill Gap 채널 아님**.

#### Rationale

- **혼합(γ) 선택 이유 (A)**: 적신호는 "형식 일치" 같은 자동 검증 가능 항목과 "적절성 판단" 같은 주관 항목이 섞여 있음. 한쪽만 택하면 다른 쪽을 포기하는 셈. 혼합이 유일하게 현실적.
- **ε 3층 구조 선택 이유 (B)**: Phase 1-a의 SSoT 원칙을 자연스럽게 연장. 체크리스트 상세 = guide (내용), skill은 포인터 (워크플로우에서 게이트 노출), AGENTS는 슬로건 (선제 트리거). 3층이 서로 역할이 달라 중복 없음.
- **PUT 직전 트리거 이유 (C)**: 작성 직후 트리거하면 작업자가 "나중에 다시 볼게"라고 미루는 경향. PUT 직전은 마지막 방어선이고, 여기서 실패해도 작업이 외부에 나가지 않음.
- **위반만 명시 이유 (D)**: 모든 산출물에 "checklist passed" 요구 시 과부하(특히 작은 commit). 암묵적 통과로 두되 **위반은 반드시 명시** — 실전 안정성 유지. DONE_WITH_CONCERNS 채널이 이미 존재하므로 재사용.
- **DONE_WITH_CONCERNS / Skill Gap 분리 이유 (E)**: 성격이 다름. 위반 = "이번 작업자가 놓침" → 본 작업 재작업. 결함 = "체크리스트 자체가 부족" → skill 체계 수정. 혼동 시 피드백 루프가 어느 쪽으로 가야 할지 불명확.
- **유형 A만 적용 이유 (F)**: Phase 1-b Decision I와 동일. 유형 B는 외부 산출물 개념이 다르거나 없음 (systematic-debugging-aim의 산출물은 코드 수정 자체). 일률 적용 시 방법론 흐름 방해.
- **자동화 명령 guide 내 병기 이유 (G)**: 작업자가 매번 수동 체크하면 누락 발생. 자동화 가능한 건 명령 한 줄로 실행 가능하게 — 실제 수행 유도.
- **적신호 누적을 guide 갱신으로 간주 이유 (H)**: 적신호 목록이 시간에 따라 성장하는 건 정상(새 실수 패턴 발견). 이 성장은 "skill 설계 결함"이 아니라 "SSoT 성숙"이므로 Skill Gap 채널 아님.

## 7. 운영 규칙

### 문서화 원칙 — 4요소 구조
각 Phase 블록은 다음 4요소로 기록한다. 독자가 "무엇을 바꿨는지"뿐 아니라 **"왜 이런 고민을 했고 어떤 근거로 결정됐는지"**를 재현할 수 있어야 한다:

1. **Problem** — 어떤 문제/고민이 있었는가
2. **Options considered** — 어떤 대안들이 검토됐는가, 각 대안의 장단점
3. **Decision** — 최종 결정 (규칙/표/템플릿)
4. **Rationale** — 그 결정을 선택한 근거 (왜 다른 옵션이 아닌가)

### 진행 규칙
- 각 Phase 완료 시마다 본 문서의 해당 섹션을 `⏳ 대기` → `✅ 완료 (날짜)` 로 갱신한다
- Phase 2/3의 `⏳ 대기` 블록은 착수 시 상기 4요소 구조로 재작성한다
- 변경 사항이 있으면 **5. 변경 대상 파일 총 목록**에 누적 기록한다
- Phase 1 메타 결정(Phase 1-a, 1-b, 1-c)은 Phase 2/3 반영 시 재확인 기준이 된다
- 사용자 승인 게이트 준수: 각 Task 착수 전 결정 초안 제시 → 피드백 → 확정 → 반영

### Specific scope 판별 원칙 (Phase 3-b에서 도입)

skill 수정 시 반영할 규칙이 **일반 원칙**인지 **해당 작업에만 적용되는 specific scope**인지 판별한다.

**판단 기준**:
- **일반 원칙** = skill이 커버하는 도메인 내 **여러 작업에 재사용** 가능. 예: "gtest canonical format", "MR Self-review checklist", "다중 파일 wording 통일"
- **Specific scope** = 특정 작업의 규모·맥락에 한정. 재사용 가능성 낮음. 예: "보류 판단 프레임(SPA 규모 작업용)"

**실행 루틴**:
1. 수정 제안이 나오면 각 항목에 대해 specific scope 여부 점검
2. 애매하면 사용자에게 상의 (판단은 사용자 권한)
3. Specific으로 판단되면 skill에서 제외 — 해당 작업의 작업 메모나 plan에만 기록
4. 기존 수정도 회고적으로 점검 가능 (Phase 3-b에서 Phase 2-a/2-b/3-a 역검토 수행)

**배경**: Phase 3-b 진행 중 사용자가 G11(보류 판단 프레임) 제외를 지시하며 "specific한 scope이 skill에 적용됐다면 다시 제외하고 싶다" 의사 표명. 이 원칙은 향후 모든 수정에 적용.

### 다른 담당자 변경 병합 — 3단 판정 원칙 (Phase 4에서 도입)

다른 담당자 또는 외부 소스의 변경을 세션 작업에 병합할 때 각 항목을 다음 3단으로 판정:

1. **중복?** — 프로젝트 내 다른 곳에 이미 동일/유사 규칙이 있는가? 있으면 생략 또는 기존 섹션 보강으로 처리
2. **충돌?** — 세션 결정(또는 기존 규칙)과 서로 모순되는가? 충돌 시 사용자 결정 필요 (내 본 유지 / 상대본 채택 / 중간)
3. **Specific?** — 해당 작업에 한정된 좁은 scope인가? Specific이면 일반 원칙으로 승격하거나 제외

**적용 순서**: 중복 → 충돌 → specific 순으로 점검하고, 해당 없으면 "적용". 어디 하나 걸리면 처리 방안 결정.

**예시 (Phase 4 적용 결과)**:
- #2 doxygen declaration: 중복(gtest에 동일 줄) → 전역 승격 + 하위 참조로 정리
- #3 값→문자열 map 재사용: specific → common 일반화
- #4 helper fallback: specific → #3과 통합
- #5 외부 helper 비공개: 중복(기존 3섹션 조합) → 기존 섹션 보강으로 대체
- #7 body 주석 완화: 충돌(내 4개 필수 vs 상대 2개 최소) → 사용자 결정 "내 본 유지"

### Branch 작업 위생 원칙 (Phase 4에서 도입)

**배경**: 세션 중 `git checkout`으로 tracked 파일 local change 유실 사고 발생.

- 세션 중 branch 전환 금지 (특히 양 branch 간 동일 파일 내용이 다른 경우). 필요 시 stash 후 checkout → stash pop 순서
- `.gitignore` 상태를 사전 인지: 비추적 파일은 branch 전환 영향 없음 (장점이자 단점)
- 긴 세션에서는 중간 commit 또는 명시적 stash로 변경 보존
- reflog 확인 습관: `git reflog -20`으로 HEAD 이동 이력 추적
